🛠️ 🔧 Scripting, Automation & Coding
1 Q1: Parse a log file and find the top 10 most frequent IP addresses Scripting L61-62

Sample Answer (Python):

from collections import Counter
import re

def top_ips(log_file, n=10):
    ip_pattern = re.compile(r'\b\d{1,3}(?:\.\d{1,3}){3}\b')
    ip_counts = Counter()
    
    with open(log_file) as f:
        for line in f:
            ips = ip_pattern.findall(line)
            ip_counts.update(ips)
    
    return ip_counts.most_common(n)
💡 Key Points: Mention streaming (line-by-line) for large files, regex efficiency, and how you'd handle IPv6. At senior level, discuss parallelization with multiprocessing for multi-GB logs.
2 Q2: Write a script to recursively find all files larger than 100MB Scripting L61-62

Bash:

find /path -type f -size +100M -exec ls -lh {} \;

PowerShell:

Get-ChildItem -Path C:\ -Recurse -File |
    Where-Object { $_.Length -gt 100MB } |
    Select-Object FullName, @{N='SizeMB';E={[math]::Round($_.Length/1MB,1)}} |
    Sort-Object SizeMB -Descending
💡 Follow-up: How would you schedule this as a monitoring job? (Cron/Task Scheduler to Azure Function to alert via Logic App or PagerDuty)
3 Q3: Design a script that monitors disk usage and auto-remediates Scripting L63-64

Approach:

  • Check disk usage with df -h or Get-PSDrive
  • If threshold >85%: purge temp files, rotate logs, notify via webhook
  • If >95%: escalate to PagerDuty, stop non-critical services
  • Log all actions for audit trail
💡 Senior angle: Discuss idempotency, dry-run mode, integration with Azure Monitor custom metrics, and how you'd test this in a staging env without real disk pressure.
4 Q4: Implement a rate limiter for an API endpoint Scripting L65+

Approach — Token Bucket / Sliding Window:

  • Token Bucket: Fixed capacity, tokens replenish at steady rate. Each request consumes a token.
  • Sliding Window: Track requests in a time window (e.g., Redis sorted set with timestamps)
  • Distributed: Use Redis INCR + EXPIRE for cluster-wide rate limiting
# Redis-based sliding window (pseudocode)
def is_allowed(user_id, limit=100, window=60):
    key = f"rate:{user_id}"
    now = time.time()
    pipe = redis.pipeline()
    pipe.zremrangebyscore(key, 0, now - window)
    pipe.zadd(key, {str(now): now})
    pipe.zcard(key)
    pipe.expire(key, window)
    _, _, count, _ = pipe.execute()
    return count <= limit
💡 Principal angle: Discuss trade-offs (memory vs accuracy), graceful degradation, per-tenant vs global limits, and how Azure API Management handles this natively.

🔄 🔄 CI/CD & Pipeline Architecture
1 Q1: What is CI vs CD vs Continuous Deployment? CI/CD L61-62
  • CI (Continuous Integration): Developers merge code frequently — automated build + unit tests run on every commit. Catches integration issues early.
  • CD (Continuous Delivery): Code is always in a deployable state. Artifacts pass through staging gates but deployment to production requires manual approval.
  • Continuous Deployment: Every change that passes all pipeline stages is automatically deployed to production — no human gate.
💡 Microsoft context: Most Microsoft teams use Continuous Delivery with ring-based deployments (Ring 0 = internal dogfood, Ring 1 = canary %, Ring 2 = broad).
2 Q2: Design a CI/CD pipeline for 15+ microservices on AKS CI/CD L63-64

Architecture:

  • Repo strategy: Polyrepo (1 repo per service) with shared pipeline templates in a central repo
  • Branching: Trunk-based development, short-lived feature branches, PR to main triggers CI
  • CI Stage: Build → Unit tests → SAST (CodeQL) → Container scan (Trivy) → Push to ACR
  • CD Stage: Helm chart update → Deploy to dev → Integration tests → Deploy to staging → Smoke tests → Production (blue-green)
  • Rollback: Automatic rollback if health check fails within 5 min; Helm rollback to previous revision
  • Observability: Each deploy emits telemetry to App Insights; deployment markers in Grafana dashboards
💡 Key differentiators: Mention pipeline templates (DRY), service mesh for canary (Istio traffic splitting), contract testing between services (Pact), and GitOps with Flux/ArgoCD.
3 Q3: How would you handle database migrations in a CI/CD pipeline? CI/CD L63-64
  • Tool: Flyway or Liquibase for versioned migrations
  • Strategy: Expand-contract pattern — add new columns/tables first (expand), deploy new code, then remove old schema (contract)
  • Pipeline integration: Migration runs as a pre-deployment step in a separate job with its own approval gate
  • Rollback: Every migration has a corresponding rollback script; tested in staging before prod
  • Zero downtime: Never rename/drop columns directly — use views or dual-write patterns
4 Q4: Design a release strategy for a platform serving 100M+ users CI/CD L65+
  • Ring-based deployment: Ring 0 (team) → Ring 1 (1% canary) → Ring 2 (10%) → Ring 3 (full)
  • Feature flags: LaunchDarkly/Azure App Configuration for gradual rollout independent of deployment
  • Bake time: Each ring has mandatory bake period (4h/12h/24h) with automated health scoring
  • Automatic promotion/rollback: Based on error rate, latency P99, and customer-reported issues (ICM integration)
  • Safe deployment practices: One-box → region-pair → global; never deploy to paired regions simultaneously
💡 Microsoft-specific: Reference Azure Safe Deployment Practices (SDP), Express V2 deployment system, and how Enterprise Platform/Azure use progressive exposure.

☁️ ☁️ Infrastructure as Code & Cloud
1 Q1: Terraform vs Bicep vs ARM Templates — when to use which? IaC L61-62
  • ARM: Native Azure JSON templates. Verbose but fully supported. Good for Azure Policy and Blueprints.
  • Bicep: DSL that compiles to ARM. Cleaner syntax, first-party Azure support, great for Azure-only shops.
  • Terraform: Multi-cloud, declarative HCL, mature ecosystem with providers for AWS/GCP/Azure/K8s. Best for hybrid or multi-cloud.
💡 Microsoft teams mostly use: Bicep for pure-Azure resources, Terraform for multi-cloud or when existing modules exist. ARM is legacy but still seen in older services.
2 Q2: Describe your Terraform workflow for a production environment IaC L63-64
  • State: Remote backend in Azure Storage with state locking (lease blob)
  • Workspaces/Environments: Separate state files per environment (dev/staging/prod) or use Terragrunt
  • PR workflow: terraform plan output posted as PR comment → reviewer approves → merge triggers terraform apply
  • Modules: Shared modules in private registry, versioned with semver
  • Drift detection: Scheduled terraform plan in CI to detect manual changes
  • Security: tfsec / checkov in pipeline, no secrets in state (use Key Vault references)
3 Q3: How do you handle Terraform state conflicts in a large team? IaC L65+
  • State locking: Backend-native locking (Azure blob lease, S3 DynamoDB)
  • State splitting: Decompose monolithic state into domain-specific states (networking, compute, data) with terraform_remote_state data sources for cross-references
  • Ownership model: Each team owns their state; shared infrastructure managed by platform team
  • Import/Move: Use terraform state mv and moved blocks for refactoring without destroy/recreate

🔐 🔒 Security & Secrets Management
1 Q1: How do you manage secrets in CI/CD pipelines? Security L61-62
  • Never: Hardcode secrets in code, pipeline YAML, or Docker images
  • Azure DevOps: Variable groups linked to Azure Key Vault; secrets auto-masked in logs
  • GitHub Actions: Repository/org secrets + OIDC for passwordless Azure auth
  • Runtime: Workload Identity (AKS) or Managed Identity for service-to-service auth
  • Rotation: Automated key rotation via Key Vault policies + Event Grid notifications
2 Q2: A secret was accidentally committed to a repo. What do you do? Security L63-64

Immediate Response (within minutes):

  • 1. Revoke: Immediately rotate/revoke the exposed credential
  • 2. Assess blast radius: Check audit logs for unauthorized usage
  • 3. Remove from history: git filter-branch or BFG Repo-Cleaner (force-push required)
  • 4. Invalidate caches: Clear CI caches, CDN caches, any mirrors

Prevention:

  • Pre-commit hooks (detect-secrets, git-secrets)
  • Pipeline gate: CredScan / GitHub secret scanning
  • Branch protection: require PR reviews
3 Q3: Design a zero-trust security model for a microservices platform Security L65+
  • Identity: Every service has a Managed Identity; no shared secrets
  • Network: Service mesh (Istio) with mTLS for all east-west traffic
  • Authorization: OPA/Gatekeeper policies for fine-grained RBAC
  • Supply chain: Signed container images (Notation/Cosign), admission controllers verify signatures
  • Data: Encryption at rest + in transit; customer-managed keys where required
  • Audit: Centralized audit trail (Azure Monitor + Sentinel SIEM)

🏗️ 🏗️ System Design
1 Q1: Design a centralized logging system ingesting TBs/day Design L63-64

Architecture:

  • Ingestion: Event Hubs (partitioned by service) → Stream Analytics or Azure Data Explorer (ADX)
  • Storage: Hot tier (ADX, 30 days) → Warm (Azure Data Lake, 1 year) → Cold (Archive blob)
  • Query: KQL in ADX for real-time; Spark in Synapse for batch analytics
  • Scale: Event Hubs auto-inflate; ADX cluster autoscale on CPU/ingestion rate
  • Cost optimization: Sampling at ingestion for verbose logs; full fidelity for errors/security events
💡 Microsoft uses: Geneva (internal) → Kusto/ADX clusters. External equivalent: Azure Monitor + Log Analytics workspace with data collection rules.
2 Q2: What happens when you type www.microsoft.com in a browser? Design L61-62
  • DNS Resolution: Browser cache → OS cache → Recursive resolver → Root → .com TLD → Authoritative NS → A/AAAA record
  • TCP + TLS: 3-way handshake → TLS 1.3 handshake (ALPN, certificate validation, session key)
  • HTTP/2 Request: GET / with headers → Azure Front Door (CDN/WAF) → Origin (App Service/AKS)
  • Server processing: Load balancer → web server → application logic → database/cache
  • Response: HTML → browser parses DOM → fetches CSS/JS/images (parallel) → renders page
3 Q3: Design a multi-region HA deployment with <5min RPO Design L65+
  • Traffic: Azure Front Door with priority routing (active-passive) or weighted (active-active)
  • Compute: AKS clusters in 2+ regions with identical Helm releases via GitOps
  • Data: Cosmos DB (multi-region writes, strong consistency for critical paths) or SQL with auto-failover groups
  • Stateless design: Session state in Redis (geo-replicated); no local disk dependencies
  • Failover testing: Monthly chaos engineering (Azure Chaos Studio) — simulate region failure
  • RPO <5min: Async replication with WAL shipping; Cosmos DB guarantees <5s with session consistency

📊 📊 Observability & Incident Response
1 Q1: Set up end-to-end monitoring for a cloud-native app Observability L63-64

Three Pillars:

  • Metrics: Azure Monitor / Prometheus → Grafana dashboards. Key metrics: latency (P50/P95/P99), error rate, throughput, saturation
  • Logs: Structured logging (JSON) → Log Analytics workspace. Correlation IDs across services.
  • Traces: OpenTelemetry SDK → Application Insights. Distributed tracing shows request flow across services.

Alerting Strategy:

  • SLO-based alerts (burn rate) over threshold-based
  • Severity mapping: P99 latency breach = Sev3; error rate >1% = Sev2; total outage = Sev1
  • Escalation: PagerDuty/ICM → on-call → bridge → incident commander
2 Q2: Walk me through handling a Sev1 production incident Observability L63-64

Incident Lifecycle:

  • Detect (TTD): Alert fires → on-call acknowledges within 5 min
  • Triage (TTA): Assess scope and customer impact → declare Sev1 if broad impact
  • Mitigate (TTM): First priority is restore service — rollback, failover, traffic shift, or hotfix
  • Communicate: Status page update, stakeholder notification, bridge call
  • Root Cause (TTR): After mitigation — deep dive, timeline, 5-whys analysis
  • Postmortem: Blameless review within 48h. Repair items tracked in backlog with owners/dates.

🌟 Your STAR Example (from Platform SRE):

S: Sev2 ICM on Triage-Data service — 40+ alerts fired simultaneously for ADF pipeline failures.
T: As on-call SRE, needed to triage, mitigate, and coordinate across teams.
A: Used Kusto queries to isolate root cause (storage account throttling), redirected traffic, engaged storage team. Created automated runbook for future.
R: TTM reduced from 45 min to 12 min for similar incidents. Runbook adopted team-wide.

3 Q3: Design an SLO/SLA framework for a platform team Observability L65+
  • SLI (Indicator): Measurable signal — e.g., "proportion of successful HTTP requests"
  • SLO (Objective): Target — e.g., 99.9% availability (allows 43 min downtime/month)
  • Error Budget: 100% - SLO = budget for experimentation/deployments. If budget exhausted → freeze deployments, focus on reliability.
  • Burn rate alerts: Alert when consuming error budget 14.4x faster than sustainable (= will exhaust in 1 hour)
  • SLA (Agreement): Customer-facing promise with financial penalties. Always set SLA < SLO (buffer).

☸️ ☸️ Kubernetes & Containers
1 Q1: Pods vs Deployments vs StatefulSets Kubernetes L61-62
  • Pod: Smallest deployable unit. One or more containers sharing network/storage. Ephemeral.
  • Deployment: Manages ReplicaSets of identical Pods. Handles rolling updates, rollbacks, scaling. Use for stateless apps.
  • StatefulSet: Like Deployment but with stable network identity and persistent storage per Pod. Use for databases.
  • DaemonSet: One Pod per node (monitoring agents, log collectors).
2 Q2: How do you implement zero-downtime deployments in AKS? Kubernetes L63-64
  • Rolling update: maxSurge: 25%, maxUnavailable: 0 — always have full capacity
  • Readiness probes: New pods only receive traffic after passing health checks
  • Pre-stop hooks: Graceful shutdown — drain connections before termination (preStop: sleep 15)
  • PDB (Pod Disruption Budget): minAvailable: 80% prevents too many pods being evicted during upgrades
  • Blue-green with Istio: VirtualService routes 100% to v1 → shift 10% to v2 → monitor → shift 100%
3 Q3: Design AKS cluster architecture for a multi-tenant SaaS platform Kubernetes L65+
  • Isolation model: Namespace-per-tenant with NetworkPolicies (deny-all default) + ResourceQuotas
  • Node pools: System pool (control plane) + User pools (per-tier) with taints/tolerations
  • Networking: Azure CNI with private cluster, egress via Azure Firewall, ingress via AGIC + WAF
  • Security: Azure Policy (OPA Gatekeeper) — no privileged containers, no latest tag, mandatory resource limits
  • Cost: Cluster autoscaler + KEDA for event-driven scaling; spot instances for batch workloads
  • Upgrades: Blue-green node pools — create new pool at N+1 version, cordon/drain old pool, delete

1 Q1: Tell me about a time you caused (or nearly caused) a production outage Behavioral All Levels

🌟 STAR Framework:

S: During a routine Terraform apply for Azure Data Factory pipelines, a misconfigured variable caused all 12 production pipelines to be destroyed and recreated.
T: Needed to restore service within SLA (30 min TTM) and prevent recurrence.
A: Immediately triggered ADF backup restoration from ARM export (15 min). Implemented prevent_destroy lifecycle rules, added terraform plan diff review as mandatory PR gate, and created a shared dangerous operations checklist.
R: Service restored in 18 min. Zero repeat incidents since. The plan-review gate caught 3 potentially destructive changes in the following month.

2 Q2: Describe a time you disagreed with a developer about a deployment approach Behavioral All Levels

🌟 STAR Framework:

S: A dev team wanted to deploy a major database schema change (column rename) during peak hours with a single big-bang migration.
T: Needed to find a compromise that met their deadline without risking customer-facing downtime.
A: Proposed an expand-contract pattern: add new column → dual-write in code → backfill → switch reads → drop old column. Presented latency data showing the big-bang approach would cause 8+ min of 500 errors.
R: Team adopted expand-contract. Migration completed across 3 deploys with zero downtime. Pattern became the team standard for all schema changes.

3 Q3: How do you prioritize when multiple Sev1/Sev2 incidents fire simultaneously? Behavioral All Levels

🌟 STAR Framework:

S: Three Sev2 ICMs fired within 10 minutes: ADF pipeline failures, App Insights alerting on elevated latency, and a certificate expiry warning.
T: As sole on-call, needed to triage and act on all three within SLA.
A: Applied impact-based triage: (1) Certificate = highest urgency (hard deadline, full outage if expired), (2) ADF = customer-visible data delay, (3) Latency = elevated but not erroring. Renewed cert first (3 min), then investigated ADF (found upstream dependency, escalated), then latency (auto-resolved after ADF fix).
R: All three resolved within SLA. Created a triage priority matrix that the team now uses for multi-incident scenarios.

4 Q4: Tell me about a time you learned something new to solve a problem Behavioral All Levels

🌟 STAR Framework (Growth Mindset):

T: I volunteered to build an AI-driven triage agent despite having no prior experience with Azure OpenAI or agentic frameworks.
A: Self-taught prompt engineering and Azure OpenAI API over 2 weeks. Built a prototype SRE Agent that classifies incoming ICMs, suggests runbooks, and auto-routes to the correct crew. Iterated based on team feedback.
R: Agent now handles initial triage for 70% of incidents, reducing human TTA from 15 min to 2 min. Presented at team all-hands; being evaluated for broader adoption.

🎯 🎯 Scenario-Based Questions
1 Q1: Your deployment pipeline takes 45 minutes. How would you optimize it? Scenario L63-64

Investigation & Optimization:

  • Measure: Add timing to each stage — identify the bottleneck
  • Parallelize: Run unit tests, linting, and security scans in parallel stages
  • Cache: Docker layer caching, dependency caching (node_modules, pip cache), Terraform provider cache
  • Test optimization: Run only affected tests (test impact analysis), move integration tests to post-deploy
  • Build optimization: Multi-stage Docker builds, distroless base images (smaller = faster push/pull)
  • Infrastructure: Self-hosted agents with warm caches, larger VM SKUs for build agents
💡 Target: CI should be <10 min for fast feedback. CD can be longer but with manual gates between stages.
2 Q2: A team keeps manually SSH-ing into production servers. How do you change this? Scenario L63-64
  • Immediate: Implement Azure Bastion (no direct SSH), require JIT access with time-limited PIM elevation
  • Short-term: Convert manual fixes into automated runbooks (Azure Automation/Ansible)
  • Medium-term: Immutable infrastructure — replace fix-in-place with redeploy-from-image
  • Culture: Make manual changes visible (audit logs in dashboard), celebrate "cattle not pets" wins
  • Guardrails: Remove standing SSH access entirely; all changes via pipeline with approval gate
3 Q3: You inherit a legacy monolith with no tests, no CI/CD. What's your 90-day plan? Scenario L65+

Week 1-2: Assess & Quick Wins

  • Map deployment process (who does what, how long, failure rate)
  • Set up basic CI: build on every commit, even without tests
  • Containerize the monolith (lift-and-shift to Docker)

Week 3-6: Foundation

  • Add smoke tests and integration tests for critical paths (80/20 rule)
  • Implement CD to staging environment with single-click prod deploy
  • Set up monitoring (App Insights, health endpoints, alerting)

Week 7-12: Maturity

  • Progressive delivery (canary deploys)
  • IaC for all infrastructure (Terraform/Bicep)
  • Begin strangler-fig pattern: extract one bounded context as a microservice
  • Document Architecture Decision Records (ADRs)
💡 Key principle: Don't rewrite everything. Incrementally improve. Ship value while improving the system. Every deployment should be boring.
