📘 Section 0: Key Definitions

This section creates a shared vocabulary before the interview drills. Use these definitions to anchor concise opening answers, then layer examples from architecture, delivery, and operations.

This section creates a shared vocabulary before the interview drills. Use these definitions to anchor concise opening answers, then layer examples from architecture, delivery, and operations.

What is Azure?

Azure is Microsoft's cloud platform with 200+ services deployed across 60+ regions. It supports IaaS, PaaS, and SaaS patterns for compute, networking, identity, data, analytics, AI, and operations.

  • IaaS example: Virtual Machines and Virtual Networks
  • PaaS example: App Service, Azure Functions, Azure SQL
  • SaaS example: Microsoft 365 and Dynamics 365

What is CI/CD?

Continuous Integration means every merge triggers automated build and test validation. Continuous Delivery/Deployment adds automated release flow so validated code moves through environments consistently.

  • CI focuses on fast feedback and broken-build prevention
  • CD standardizes promotion, approvals, rollout, and rollback
  • Net effect: fewer manual errors and faster release cycles

What is DevOps?

DevOps is a culture and engineering practice model that unifies development and operations to improve flow, resilience, and feedback. A useful frame is CALMS: Culture, Automation, Lean, Measurement, and Sharing.

  • Goal: shorter delivery cycles with reliable production outcomes
  • Success signals: lower change failure rate, faster recovery, higher deployment frequency

What is Azure DevOps?

Azure DevOps is Microsoft's integrated software delivery platform spanning source control, planning, testing, packaging, and pipelines.

  • Azure Repos for Git repositories
  • Azure Pipelines for CI/CD
  • Azure Boards for work tracking
  • Test Plans and Artifacts for quality and packages

What is Git?

Git is the distributed version control system created by Linus Torvalds. Every clone has full history, which makes branching, merging, and offline work fast.

  • Core ideas: commits, branches, merge or rebase, and remotes
  • Teams use Git to audit changes, collaborate safely, and recover quickly

What is GitHub?

GitHub is Microsoft's cloud platform for hosting Git repositories and surrounding collaboration workflows.

  • Pull requests, reviews, Issues, Discussions, and Projects
  • GitHub Actions, Copilot, Pages, Dependabot, and Advanced Security

What is GitLab?

GitLab is an all-in-one DevOps platform that combines source control, CI/CD, package storage, security scanning, and planning in one product experience.

  • Strong built-in pipeline and security posture
  • Often chosen when teams want a single integrated DevSecOps suite

What is GitHub Actions?

GitHub Actions is GitHub's CI/CD capability. Workflows live in .github/workflows/ and are composed of events, jobs, steps, reusable actions, and environment controls.

  • Trigger on push, PR, schedule, release, or manual dispatch
  • Use marketplace actions or private reusable workflows

1 Q1. App Service vs Azure Functions — when do you choose each? Azure Foundation

Decision framework

ChooseBest forWhy
App ServiceAlways-on web apps, APIs, admin portals, predictable trafficLong-running HTTP workloads, custom startup, stable hosting model, easy deployment slots
Azure FunctionsEvent-driven automation, bursty traffic, background jobs, glue codeScales by trigger volume, pay-per-execution model, many triggers/bindings
  • Use App Service when the application owns routing, middleware, sessionless APIs, and sustained requests.
  • Use Functions when business logic is triggered by queue messages, timers, storage events, or lightweight HTTP endpoints.
  • Latency-sensitive front-door APIs usually fit App Service better because cold starts are easier to avoid.
  • Back-office workflows, fraud scoring, notifications, and ETL fan-out often fit Functions.

2 Q2. Explain Azure Functions triggers and bindings using a fraud detection example. Azure Intermediate

Real-world framing

  • Trigger starts the function: HTTP, Service Bus, Event Hub, Blob, Queue, Cosmos DB change feed, or Timer.
  • Input bindings hydrate related data without boilerplate connection code.
  • Output bindings push results to destinations such as Cosmos DB, Service Bus, or Storage.

For fraud detection, an Event Hub trigger can ingest transactions, pull device history from Cosmos DB, and emit high-risk findings to Service Bus for analyst workflows.

public class FraudScreenFunction
                                                            {
                                                                [FunctionName("FraudScreenFunction")]
                                                                public static async Task Run(
                                                                    [EventHubTrigger("txn-stream", Connection = "EventHubConnection")] string transactionJson,
                                                                    [CosmosDB(
                                                                        databaseName: "risk",
                                                                        containerName: "deviceProfiles",
                                                                        Connection = "CosmosConnection",
                                                                        SqlQuery = "SELECT * FROM c WHERE c.deviceId = {deviceId}")]
                                                                    IEnumerable<DeviceProfile> priorProfiles,
                                                                    [ServiceBus("high-risk-alerts", Connection = "ServiceBusConnection")]
                                                                    IAsyncCollector<FraudAlert> alerts,
                                                                    ILogger log)
                                                                {
                                                                    var score = FraudRules.Calculate(transactionJson, priorProfiles);
                                                                    if (score >= 80)
                                                                    {
                                                                        await alerts.AddAsync(new FraudAlert { Severity = "High", Score = score });
                                                                    }
                                                                    log.LogInformation($"Fraud score: {score}");
                                                                }
                                                            }
  • Bindings reduce plumbing, but for complex retries, transactions, or batching, the SDK may still be preferable.
  • Idempotency matters: duplicate events can happen, so use event IDs or dedupe keys.
Hands-on tip: Build a small Function locally with an HTTP trigger and a Queue output binding, then add an idempotency key to prevent duplicate processing.

5 Q5. How do you secure Azure Functions in production? Azure Architect

Production security checklist

  • Identity: prefer managed identity over connection strings or embedded secrets.
  • Secrets: store in Azure Key Vault and reference them through managed identity.
  • Networking: private endpoints, VNet integration, IP restrictions, and APIM in front of public endpoints when needed.
  • Authorization: Entra ID auth for internal APIs; function keys alone are not enough for sensitive workloads.
  • Compliance: enforce 1ES-aligned scans such as CredScan, BinSkim, CodeQL, dependency scanning, and branch protection.
  • Observability: structured logs, secret redaction, and alerting on auth failures or elevated error rates.

7 Q7. What branch policies and PR validation would you set in Azure DevOps? CI/CD Intermediate

Recommended branch governance

  • Require PRs into main; block direct pushes.
  • Require successful build validation, status checks, and linked work items where needed.
  • Minimum two reviewers for risky areas; optional code owners or path-based reviewers.
  • Require comments to be resolved before completion.
  • Prefer squash merge for short-lived feature branches to keep history readable.

11 Q11. Describe a trunk-based Git branching strategy with feature branches. Git Foundation
  • Main stays releasable at all times.
  • Developers create short-lived feature branches from main.
  • PRs should be small, validated by CI, and merged quickly.
  • Use feature flags when work spans multiple merges but should stay dark in production.
git checkout main
                                                            git pull origin main
                                                            git checkout -b feature/device-risk-cache
                                                            
                                                            # work, commit, push
                                                            git add .
                                                            git commit -m "Add device risk cache adapter"
                                                            git push -u origin feature/device-risk-cache
                                                            
                                                            # keep branch fresh
                                                            git fetch origin
                                                            git rebase origin/main

Why interviewers like this answer: it balances flow speed, low merge pain, and production safety.

Hands-on tip: Simulate a one-day feature branch and explain when you would rebase, when you would merge, and when you would split the work behind a feature flag.
12 Q12. How do you resolve complex merge conflicts effectively? Git Advanced

Workflow

  1. Understand both branches before editing conflict markers.
  2. Use git rerere to remember repeated conflict resolutions.
  3. Use a visual merge tool when files have broad structural overlap.
  4. Prefer rebase for linear cleanup on private branches; avoid rewriting shared history without coordination.
git config --global rerere.enabled true
                                                            git fetch origin
                                                            git rebase origin/main

                                                            git mergetool

13 Q13. Write a fraud detection query using CTEs, window functions, and JOINs. SQL Server Advanced

What the query should prove

  • Find repeated high-value attempts from the same device or account
  • Use windows to compare recent behavior
  • Join reference data such as allow-lists, prior chargebacks, or country risk
WITH recent_txn AS (
                                                                SELECT
                                                                    t.TransactionId,
                                                                    t.AccountId,
                                                                    t.DeviceId,
                                                                    t.Amount,
                                                                    t.CountryCode,
                                                                    t.CreatedUtc,
                                                                    ROW_NUMBER() OVER (
                                                                        PARTITION BY t.AccountId
                                                                        ORDER BY t.CreatedUtc DESC
                                                                    ) AS rn,
                                                                    COUNT(*) OVER (
                                                                        PARTITION BY t.DeviceId
                                                                        ORDER BY t.CreatedUtc
                                                                        ROWS BETWEEN 9 PRECEDING AND CURRENT ROW
                                                                    ) AS last_10_attempts
                                                                FROM dbo.Transactions t
                                                                WHERE t.CreatedUtc >= DATEADD(HOUR, -24, SYSUTCDATETIME())
                                                            ),
                                                            chargeback_accounts AS (
                                                                SELECT DISTINCT AccountId
                                                                FROM dbo.Cases
                                                                WHERE CaseType = 'Chargeback'
                                                            )
                                                            SELECT
                                                                r.TransactionId,
                                                                r.AccountId,
                                                                r.DeviceId,
                                                                r.Amount,
                                                                r.CountryCode,
                                                                r.last_10_attempts,
                                                                CASE
                                                                    WHEN cb.AccountId IS NOT NULL THEN 'High'
                                                                    WHEN r.Amount >= 500 AND r.last_10_attempts >= 4 THEN 'High'
                                                                    WHEN cr.RiskTier = 'Elevated' THEN 'Review'
                                                                    ELSE 'Normal'
                                                                END AS RiskDecision
                                                            FROM recent_txn r
                                                            LEFT JOIN chargeback_accounts cb
                                                                ON cb.AccountId = r.AccountId
                                                            LEFT JOIN dbo.CountryRisk cr
                                                                ON cr.CountryCode = r.CountryCode
                                                            WHERE r.rn <= 5
                                                            ORDER BY r.CreatedUtc DESC;
Hands-on tip: Load sample transaction data and add one more fraud rule, such as impossible travel or device-sharing across many accounts.

15 Q15. How do you optimize a slow stored procedure? SQL Server Architect
  1. Get the actual execution plan and identify the expensive operator.
  2. Check cardinality estimate errors, missing indexes, spills, scans, and parameter sniffing symptoms.
  3. Validate statistics freshness and data skew.
  4. Rewrite predicates to stay SARGable and reduce row width early.
CREATE OR ALTER PROCEDURE dbo.GetRecentHighRisk
                                                                @AccountId bigint = NULL
                                                            AS
                                                            BEGIN
                                                                SET NOCOUNT ON;
                                                            
                                                                SELECT TOP (100)
                                                                    TransactionId,
                                                                    AccountId,
                                                                    Amount,
                                                                    RiskScore,
                                                                    CreatedUtc
                                                                FROM dbo.Transactions
                                                                WHERE (@AccountId IS NULL OR AccountId = @AccountId)
                                                                  AND RiskScore >= 80
                                                                ORDER BY CreatedUtc DESC
                                                                OPTION (RECOMPILE);  -- use selectively when parameter skew is severe
                                                            END;
-- Useful diagnostics
                                                            SET STATISTICS IO, TIME ON;
                                                            EXEC dbo.GetRecentHighRisk @AccountId = 101;
                                                            EXEC dbo.GetRecentHighRisk @AccountId = NULL;
                                                            SET STATISTICS IO, TIME OFF;

Discuss parameter sniffing explicitly: good for many procedures, painful when data distribution is highly skewed.

Hands-on tip: Take one slow proc, compare estimated and actual rows, then decide whether the right fix is indexing, rewrite, statistics, or parameter-handling strategy.
16 Q16. Build a monthly trend report with LAG, running totals, and month-over-month percentage. SQL Server Advanced 
WITH monthly AS (
                                                                SELECT
                                                                    DATEFROMPARTS(YEAR(CreatedUtc), MONTH(CreatedUtc), 1) AS MonthStart,
                                                                    COUNT(*) AS TicketCount,
                                                                    SUM(CASE WHEN Severity = 'High' THEN 1 ELSE 0 END) AS HighSeverityCount
                                                                FROM dbo.SupportTickets
                                                                WHERE CreatedUtc >= DATEADD(MONTH, -12, SYSUTCDATETIME())
                                                                GROUP BY DATEFROMPARTS(YEAR(CreatedUtc), MONTH(CreatedUtc), 1)
                                                            )
                                                            SELECT
                                                                MonthStart,
                                                                TicketCount,
                                                                HighSeverityCount,
                                                                SUM(TicketCount) OVER (ORDER BY MonthStart) AS RunningTotal,
                                                                LAG(TicketCount) OVER (ORDER BY MonthStart) AS PriorMonthCount,
                                                                CASE
                                                                    WHEN LAG(TicketCount) OVER (ORDER BY MonthStart) IS NULL THEN NULL
                                                                    WHEN LAG(TicketCount) OVER (ORDER BY MonthStart) = 0 THEN NULL
                                                                    ELSE ROUND(
                                                                        (TicketCount - LAG(TicketCount) OVER (ORDER BY MonthStart)) * 100.0
                                                                        / LAG(TicketCount) OVER (ORDER BY MonthStart), 2
                                                                    )
                                                                END AS MoMPercent
                                                            FROM monthly
                                                            ORDER BY MonthStart;
  • LAG compares current month with previous month.
  • SUM OVER gives the running total without self-joins.
  • Wrap divide-by-zero cases safely.
Hands-on tip: Add a rolling three-month average to the query and explain why leaders often prefer trend smoothing over single-month spikes.

19 Q19. Explain Cosmos DB consistency levels. Which would you choose for fraud detection? Cosmos DB Advanced
  • Strong: most consistent, highest coordination cost
  • Bounded Staleness: reads lag by a bounded number of versions or time interval
  • Session: read-your-own-writes per client session
  • Consistent Prefix: ordered updates without gaps, but may be stale
  • Eventual: fastest and loosest consistency

For distributed fraud detection, Bounded Staleness is often the balanced answer: it limits stale reads while preserving better global performance than Strong consistency across regions.

// Example concept

                                                            {

                                                            };

Why not Session only? Because multiple services may read the same device/account state, not just the original writer session.

Hands-on tip: Write down one scenario where Strong consistency is worth the cost and one where Eventual is acceptable. That contrast sharpens your interview answer.
